# Red-Team Review — Blindspots Found in the Draft Plan

Three independent Opus reviewers attacked the draft plan from distinct angles. This is the raw record; the final plan (`00-OVERALL-APPROACH.md`) incorporates the fixes.


## Reviewer 1 — feasibility & scope

**Verdict:** The plan is genuinely good — better-scoped than most 2-week solo POCs, with explicit MVP fallbacks per day and a "fewest moving parts" through-line. It is NOT over-scoped on the ML side (Qwen3-1.7B + LoRA + DPO on 5k rows is right-sized). The real feasibility risk is concentrated in two places the plan underestimates: (1) the back half (Days 8-14) crams the four hardest, most failure-prone, first-time-for-this-learner tasks — RunPod serverless deploy, Docker, CI/CD-with-deploy, AND a 20-page report — into 6 half-time days, with no slack for the cold-start/version/cloud debugging that historically eats whole days; (2) the timeline is expressed in elapsed days but the work is "half-time," so ~14 calendar days is really ~7 working days for a pipeline most pros would not finish in 7. The single most important change: treat Day 8 (a live endpoint, however ugly) as a Day-5/6 hard gate and pull the report to start in parallel from Day 1 as a running log, not a Day-13 sprint. Everything else is a tweak. Concretely verified gotchas below (Kaggle 12h-session cap, RunPod true-cold-start ≈2 min, Unsloth/transformers version blocklist) are real and bigger than the draft implies, but each has a cheap fix.


### [HIGH] The day-by-day plan is stated as ~14 days but explicitly assumes 'half-time effort,' so it is really ~7 full working days for SFT+DPO+merge+serve+Docker+CI/CD+eval+20-page report. The back half (Days 8-14) front-loads ALL the unfamiliar cloud/devops tasks (RunPod serverless, Docker, CI/CD-with-deploy, report) into 6 half-days with effectively zero buffer — Day 14 'buffer' is the only slack and it is already spoken for (re-run demo, fix README, final report pass). One bad cold-start debugging session or one CI/CD deploy-secret rabbit hole eats the entire buffer.
- **Why it matters:** For a learner doing RunPod/Docker/GH-Actions-deploy for the first time, any single one of these realistically costs 1-2 full days of debugging (auth, secrets, image size, endpoint config). With no slack, the report — the deliverable that most directly determines pass/fail and is hardest to fake last-minute — gets compressed into a panic on Days 13-14 and comes out thin.
- **Fix:** Re-baseline the schedule as ~7 working days with named buffer, not 14 calendar days, and make ONE structural change: start the report on Day 1 as a running build-log (paste decisions, screenshots, error messages, metrics as they happen into a single markdown file). The 20 pages then become editing, not writing. Also explicitly protect 2 full days of buffer between 'endpoint live' and 'report due,' not the current single shared Day 14.

### [HIGH] RunPod serverless cold start is presented as '~5-15s with model caching,' but the verified reality after TRUE scale-to-zero (the whole point of 60s idle timeout for cost) is much worse: users report ~2 minutes when the worker must re-pull/re-load weights, and aggressive scale-to-zero causes re-initialization within ~1 minute of inactivity. The 10-15s figure is FlashBoot 'warm revival,' which only applies under consistent traffic — exactly NOT a student demo idle 95% of the time. The plan's cost model and the 'send one warm-up request' advice both assume the good case.
- **Why it matters:** Day 8's checkpoint ('curl gets a triage response from the live endpoint') and Day 11's p95 latency metric will look alarming (first request 60-120s, timeouts likely on default client settings), which reads as 'broken' to the learner and burns debugging time chasing a non-bug. A grader hitting a cold endpoint sees a 2-minute hang.
- **Fix:** Two cheap mitigations, both documented honestly: (1) for grading/demo, set min-workers=1 or 'active workers=1' for the ~1-hour review window (costs cents/hour, scale back to zero after) rather than relying on cold-start magic; (2) in the eval harness and README, send 2-3 warm-up requests and explicitly report 'cold-start latency (first request): ~X s' SEPARATELY from 'warm p95.' This turns the ugly number into an honest, expected POC finding instead of a fire drill.

### [MEDIUM] The Unsloth version-fragility risk is real but the plan under-mitigates it. Verified: unsloth-zoo carries a long EXPLICIT blocklist of incompatible transformers versions (e.g. !=4.52.0-4.55.1, !=4.57.4/.5, !=5.0.0/5.1.0, plus an upper bound), and on T4 specifically custom Triton kernels can take 'longer than normal' to compile. 'Copy the pins from the current official notebook' helps, but the failure mode on Kaggle is a fresh image that auto-installs a transformers version inside one of these holes, breaking on import AFTER you've spent quota.
- **Why it matters:** The Day-4 smoke test is the right idea, but if Unsloth install/compile churns for >30 min (plausible on T4 kernel compile), the plan's fallback is 'switch to plain TRL+PEFT' — which itself needs its own debug pass. Losing Day 4 to dependency hell pushes everything right into the already-tight back half.
- **Fix:** Make the Day-4 smoke test the FIRST thing on Day 1 (5 min of GPU quota: just `import unsloth; FastLanguageModel.from_pretrained(...)` on 4-bit Qwen3-1.7B, run 5 training steps). You learn within 30 min of Day 1 whether Unsloth works on the current Kaggle image, so the TRL fallback decision happens with 13 days left, not 10. Pin transformers explicitly and `pip install` it AFTER unsloth to override Kaggle's preinstalled version; verify the installed version is not in the blocklist with one assert.

### [MEDIUM] Kaggle's 12-hour-per-SESSION cap (separate from the 30h/week quota) is not addressed, and the plan's '3 epochs on 5,000 rows' SFT has no time estimate validated against it. The '~3-4h SFT' is plausible for 1 epoch but 3 epochs on 5k sequence-packed bilingual rows could approach or exceed comfortable session margins, and Kaggle kills the session at 12h regardless of quota remaining — losing the run if checkpointing isn't airtight.
- **Why it matters:** A killed session mid-epoch with mis-configured save_steps loses hours of quota (30h/week is only ~2-3 full training attempts). The plan has save_steps=50 which is good, but doesn't mention resuming-from-checkpoint, pushing checkpoints to HF Hub mid-run, or that Kaggle's disk (~20GB writable, /kaggle/working ~20GB) plus the model cache can fill up.
- **Fix:** Default to 1-2 epochs for SFT (3 is overkill for a POC and raises overfitting risk on 5k rows anyway — the plan even lists '1-2 epochs' as the fallback; make it the default). Push the LoRA adapter to HF Hub at the end of every save_steps interval (resume_from_checkpoint + push), so a killed session never costs more than 50 steps. Add a one-line disk check (`df -h /kaggle/working`) to the Day-4 smoke test.

### [MEDIUM] Day 10 CI/CD includes a 'deploy step (community action or RunPod GitHub integration)' that auto-updates the RunPod endpoint. Auto-deploy-to-serverless from GH Actions is the single most fragile, least-documented link in the chain (API tokens as secrets, endpoint-update API surface, image pull from GHCR into RunPod, version mismatch). This is over-engineered for a POC where the deliverable is 'a CI/CD pipeline exists,' not 'continuous deployment to GPU serverless works flawlessly.'
- **Why it matters:** Chasing a green end-to-end auto-deploy can silently consume Day 10 entirely and bleed into the report days, for a feature no grader of a 2-week POC actually requires to be automated.
- **Fix:** Make the CI/CD deliverable lint(ruff)+test(pytest, mocked vLLM)+build+push-to-GHCR — that is a complete, demonstrable, green CI/CD pipeline. Make the RunPod redeploy a MANUAL documented step (a one-line script + screenshot in the report), exactly as the plan's own Day-10 fallback already states. Adopt the fallback as the PLAN, not the contingency. Saves a near-certain rabbit hole at zero deliverable cost.

### [MEDIUM] The ~50-100 hand-written safety preference pairs (Decision C) plus ~30 hand-written FR triage vignettes plus ~20 adversarial probes plus the heuristic-urgency labelling of the 500-row triage slice is a non-trivial amount of careful, domain-specific MANUAL authoring that the timeline hides inside 'Day 2 / Day 6 / eval-set-built-once.' Writing clinically-plausible bilingual red-flag pairs and reference urgency labels is slow, judgment-heavy work, easily 4-8 hours, and it is on the critical path for both the DPO safety story and the eval harness.
- **Why it matters:** If this gets rushed, the DPO safety pairs are low-quality (DPO won't measurably shift behavior, undermining Decision C's whole rationale) and the eval reference labels are noisy (making the headline 'emergency recall ≥90%' number meaningless). These hand-built sets are the credibility core of the safety story the report leans on.
- **Fix:** Cut the counts to the minimum that still demonstrates the technique: ~30 safety preference pairs (5-6 red-flag scenarios x a few phrasings) and ~15 triage vignettes are enough to show the method and produce a metric for a POC. Template them aggressively (one canonical 'chosen' escalation skeleton, slot in the symptom) rather than authoring each from scratch. Reuse the SAME red-flag scenarios across DPO pairs, vignettes, and adversarial probes so the authoring effort is amortized.

### [MEDIUM] Merging LoRA into 16-bit and the vLLM sanity-load are casually placed on Day 7 with a 'sanity-load with a local/Colab vllm serve' checkpoint, but the learner does not have a local GPU, so the only place to verify the merged model actually loads in vLLM before RunPod is Colab/Kaggle — and Kaggle notebooks don't easily run `vllm serve` as a server (no persistent port). This 'sanity check' may itself be a half-day of environment friction, and if skipped, a broken merge isn't discovered until Day 8 on RunPod where debugging is far harder.
- **Why it matters:** Discovering a bad merge (wrong dtype, chat-template not saved, tokenizer mismatch) for the first time inside RunPod serverless — where logs are slow and iteration is minutes per attempt — can burn Day 8 and the only buffer. The chat-template/eos-token handling for Qwen3-Base is exactly the kind of thing that silently breaks at serve time.
- **Fix:** Verify the merged model with vLLM's OFFLINE engine (`from vllm import LLM; LLM(path).generate(...)`) inside the same Kaggle/Colab notebook right after merging — no server/port needed, runs in a cell, catches dtype/tokenizer/chat-template breakage immediately. Explicitly assert the tokenizer_config.json with the ChatML template and eos `<|im_end|>` is present in the merged folder before pushing to Hub.

### [LOW] Stage 0 data is budgeted at '~1 day / CPU-only / minutes to run,' but real dataset wrangling (schema surprises in MediQAl configs, MedQuAD card/license ambiguity already flagged, UltraMedical test-split schema mismatch already flagged, ChatML templating bugs, language-mix balancing, dedup against the DPO set) is a classic hidden time sink. The plan itself lists FOUR known data caveats — each is a small debugging session. 'Minutes to run' is true for the final ETL; getting the ETL correct is not.
- **Why it matters:** Data is the input to everything downstream; a templating bug (e.g. loss computed on prompt tokens, wrong eos, FR/EN system prompt missing) silently degrades SFT and isn't caught until generation looks wrong on Day 5, forcing a re-run that costs GPU quota. The plan's own list of dataset gotchas signals this is 1.5-2 days, not 1.
- **Fix:** Budget 1.5-2 days for data and front-load the riskiest verification: on Day 2, after building 50 rows, decode 5 fully-templated examples back through the tokenizer and eyeball that the assistant span (and only it) is the label, eos is correct, and the system prompt is present. Catching the template bug on 50 rows costs minutes; catching it post-SFT costs a training run.

## Reviewer 2 — data validity / clinical safety / GDPR

**Verdict:** The plan is impressively well-organized, honest about scope, and the engineering spine (Unsloth LoRA SFT -> DPO -> merge -> vLLM on RunPod serverless -> CI) is sound and proportionate for a 2-week solo POC. But through a data-validity / clinical-safety / GDPR lens it has one blocker and a cluster of over-claims, located exactly where the project says it is strongest. The single most important change: REMOVE MIETIC. It is PhysioNet *credentialed restricted* MIMIC-IV-ED data (verified: PhysioNet Credentialed Health Data License 1.5.0, credentialing + no-redistribution), not the CC-BY-NC-SA the plan and the HF re-upload claim; baking it into a publicly-published 'GDPR-compliant' dataset and model weights directly violates the DUA and detonates the project's core 'no real patient data, GDPR-by-construction' narrative. Rebuild the triage slice from MediQAl 'Urgences' (CC-BY-4.0, exam-derived) plus a few hand-written synthetic vignettes — cleaner provenance and a stronger GDPR story. Secondarily, stop letting metrics over-promise: heuristic urgency labels make 'triage agreement' circular, English GPT-4-scored DPO does not teach clinical quality, and a ~100-case author-written harness cannot back '100% disclaimer / >=90% emergency recall' CI gates. Each has a cheap, honest fix (independent hand-labelled eval vignettes; bilingual safety-weighted DPO framed as behavior-shaping not quality; manual safety review of every emergency probe). Get those right and the POC is not just passable but exemplary on the rigor dimensions it's graded on. One piece of good news the plan needlessly worried about: Qwen3-1.7B-Base DOES ship a full ChatML chat_template — but its configured eos is <|endoftext|>, so the <|im_end|> eos/stop-token handling is a real foot-gun to nail down before serving.


### [BLOCKER] The MIETIC triage slice is built on PhysioNet *credentialed restricted* data, not CC-BY-NC-SA. The HF copy (jackf7499/MIMIC-IV-Ext_Triage_Instruction_Corpus) self-labels 'cc-by-nc-sa-4.0', but MIETIC's authoritative license (physionet.org/content/mietic/view-license/1.0.0/) is the PhysioNet Credentialed Health Data License 1.5.0: requires CITI/HIPAA credentialing, forbids sharing access with third parties, and says derivatives must be redistributed only on PhysioNet under the same DUA. The HF re-uploader had no right to relicense it. Verified June 2026.
- **Why it matters:** This is the single most damaging item for a GDPR-themed deliverable. The flagship claim ('GDPR-compliant by construction, no real patient data') becomes self-contradicting: MIMIC-IV-ED is de-identified *real ED patient records*, and you'd be (a) using credentialed data without credentialing, (b) re-distributing it inside your own public ~5000-row dataset (Deliverable 1) and baking it into publicly-pushed model weights (Deliverable 2), both of which the DUA prohibits. An OC reviewer who clicks one link sees the project's data-governance story collapse precisely where it claims to be strongest.
- **Fix:** Drop MIETIC entirely. Build the ~300-500 row triage slice ONLY from MediQAl 'Urgences' rows (CC-BY-4.0, exam-derived, no patient data) template-wrapped into the 3-part triage format. You lose 'English real-ED grounding' but gain clean, honest provenance. For EN triage examples, hand-write ~30-50 synthetic vignettes (no real patients) instead. State in the report: 'We deliberately excluded MIMIC-derived triage corpora because their PhysioNet credentialed license forbids redistribution and is incompatible with an openly-published POC dataset' — that sentence turns a blocker into evidence of GDPR maturity. (Confidence: high.)

### [HIGH] The QA->triage gap is papered over, not closed. The recipe produces a bilingual medical-QA model and attaches heuristic urgency labels derived from MediQAl's 'medical_subject' field + question type. Urgency is being *invented* from topic, not from clinical severity. A question filed under 'Urgences' can be a benign exam item; a deadly presentation (silent MI, sepsis) can sit under 'Cardiologie' or 'Infectiologie'. The model learns topic->label, not symptom->severity.
- **Why it matters:** The eval then measures 'triage agreement' against these same heuristic labels, so it grades the model against a fiction it was trained on — circular validity. The report could claim 'X% triage agreement' that means nothing clinically. This is the 'QA bot wearing a triage label' failure the brief warns about.
- **Fix:** Keep the slice tiny and reframe it honestly as an 'urgency-routing *style* demonstration', not triage accuracy. For the EVAL specifically, do NOT reuse heuristic labels: have the ~30 FR triage vignettes carry urgency labels *you assign by hand from the symptom description*, independent of training-time heuristics, so train and test labels come from different processes. Report triage agreement only on those hand-labelled vignettes plus a one-line caveat: 'urgency labels are author-assigned, not clinically validated; this measures behavioral consistency, not triage safety.' (Confidence: high.)

### [HIGH] DPO with UltraMedical-Preference does not teach *clinical quality* — it teaches GPT-4's stylistic preferences in English. The pairs are GPT-4-scored (the dataset card itself admits self-preference bias toward GPT-4 outputs), English-only, and chosen/rejected differ mostly in completeness/formatting, not clinical correctness. None of it is French, none of it is triage.
- **Why it matters:** You will fine-tune a FR-heavy (82%) model's alignment on 100% English preference signal scored by a model with known self-bias, then claim DPO 'nudges toward safer, more complete answers.' At best it's a no-op on French; at worst it degrades French fluency (the very risk the plan flags) and imports GPT-4 verbosity, not safety. The 'DPO teaches clinical quality' claim is not supported by this data.
- **Fix:** Right-size the claim: DPO here is a *technique demonstration* (the graded deliverable) plus a *safety-behavior* lever via your hand-written safety pairs — that's the part that genuinely matters. Make the safety pairs bilingual (half FR) and a larger fraction of the mix (e.g. 300-500 of ~1500 total rather than 50-100 of 3000), and subsample UltraMedical to ~1000. Report DPO success as 'shifted escalation/disclaimer behavior on the safety probe set', NOT as 'improved clinical quality'. (Confidence: high.)

### [HIGH] Safety eval thresholds (100% disclaimer, >=90% emergency recall, 0% danger patterns) are asserted as CI gates but the harness is ~100 cases with ~20 adversarial probes, all author-written. 90% recall on ~10-15 emergency vignettes means missing 1 still passes. A regex 'danger check' cannot detect subtly wrong dosing or a plausible-but-fatal reassurance.
- **Why it matters:** A 1.7B base-derived model WILL occasionally produce confident, wrong, dangerous advice, and an author-written 20-probe set graded by regex will not catch the realistic failure modes. Framing these as 'CI gates' implies a safety guarantee the harness cannot back. If demoed live and it misses an MI, the over-claim is exposed.
- **Fix:** Demote the thresholds from 'CI gates' to 'reported metrics + a manual safety review of every emergency-probe output.' Keep the regex check but pair it with a short hand-graded rubric (escalated? disclaimer present? no diagnosis/dose? plausible-but-wrong?) on all ~20 probes. State the honest limitation in the report: 'safety verified on a small author-written probe set; not a clinical validation.' POC-proportionate and far more credible than a fake 100%/90% gate. (Confidence: high.)

### [MEDIUM] The 'run Presidio on already-de-identified data and find ~zero PII = the correct expected result' framing is honest for *exam-derived* sources but becomes dishonest if MIETIC stays in (real de-identified ED notes can carry residual quasi-identifiers: shifted dates, ages>89, rare conditions, free-text leakage). The plan asserts near-zero findings as a foregone conclusion *before running the audit*.
- **Why it matters:** If you predeclare 'we expect ~0 findings' and Presidio on MIMIC-derived rows surfaces dates/ages, you either suppress it (dishonest) or your 'zero PII expected' narrative looks naive. Asserting the outcome before running the audit is methodologically backwards for a GDPR deliverable.
- **Fix:** Two-part: (1) Removing MIETIC (see blocker) makes near-zero PII a defensible genuine expectation. (2) Reframe the audit as a *test with a hypothesis*, not a foregone conclusion: 'Sources are exam/public text, so we hypothesize minimal PII; we ran Presidio to verify and report actual findings.' Report real numbers whatever they are. Cite GDPR Recital 26 for exam-derived data, but never claim anonymity for any patient-derived source. (Confidence: high.)

### [MEDIUM] eos_token misconfiguration risk. The plan instructs eos_token='<|im_end|>'. Verified against tokenizer_config.json: Qwen3-1.7B-Base ships a full ChatML chat_template AND both <|im_start|>/<|im_end|> tokens (so the plan's worry that Base lacks a template is unfounded — good news), but its configured eos_token is <|endoftext|>, not <|im_end|>. The eos swap is therefore a real foot-gun.
- **Why it matters:** If you train on ChatML (assistant turns ending in <|im_end|>) but generation/serving still treats <|endoftext|> as eos, the model may not stop cleanly, or vLLM may not register <|im_end|> as a stop token -> runaway generations / broken triage outputs in the live demo. A classic silent failure that survives a short-max_steps Day-4 smoke test and bites at serving.
- **Fix:** Be explicit and consistent: train with <|im_end|> as the assistant-turn terminator, set it as eos in the training config, AND pass it as a stop token to vLLM (stop=['<|im_end|>'] / correct eos_token_id in the serving config). Add an assertion in the Day-7 vLLM sanity load: generate one triage answer and confirm it terminates at <|im_end|> with no trailing garbage. Simplest path: reuse unsloth/Qwen3-1.7B-Base's documented template/config rather than hand-rolling. (Confidence: high.)

### [MEDIUM] Starting from -Base (not -Instruct) is sold as 'you own the format with no conflicting alignment', but for a 2-week clinical-safety POC it raises both safety and effort cost. A base model has NO instruction-following, NO refusal behavior, NO safety priors. All safety must be installed by your ~5k SFT + tiny DPO + system prompt; with a FR-heavy QA corpus and ~100 safety pairs the model has little reason to refuse or escalate outside the exact patterns it saw.
- **Why it matters:** For a *medical* assistant, an Instruct model already declines to diagnose/prescribe and has guardrails you'd otherwise recreate from scratch in 2 weeks. Choosing Base trades a safety head-start for a 'cleaner narrative' — the wrong trade for a clinical-safety-graded POC. The plan never weighs this safety cost against the format-purity benefit.
- **Fix:** Reconsider Qwen3-1.7B (Instruct) as the SFT base: still Apache-2.0, still fits the T4, keeps inherited refusal/safety behavior, and SFT adapts format/domain on top. If you keep -Base for the learning narrative, over-weight safety in the SFT data (more safety examples, not just DPO pairs) and say so. Either way, explicitly document the Base-vs-Instruct safety tradeoff — reviewers reward seeing it named. (Confidence: medium.)

### [MEDIUM] MCQA accuracy is in the metrics table and will anchor the 'before/after fine-tuning' story, but MCQA accuracy measures medical-exam knowledge, not triage quality, and a 1.7B model LoRA-tuned on ~5k rows can show flat or *negative* MCQA movement vs base (domain/format adaptation often trades benchmark accuracy for format adherence).
- **Why it matters:** The report's headline 'fine-tuning improved things' could be contradicted by the one hard number (MCQA) going down, forcing an awkward explanation or a quiet omission. And even if it rises, it's the wrong proxy — it says nothing about whether the thing triages safely.
- **Fix:** Lead the eval with *behavioral* metrics fine-tuning genuinely changes (language-match rate, disclaimer presence, escalation on red-flag probes, format adherence) and present MCQA as a secondary 'sanity check that we didn't destroy medical knowledge.' Pre-commit to reporting MCQA honestly even if flat/down, framed as 'expected: domain/format adaptation, not benchmark maximization.' Protects the narrative whichever way the number moves. (Confidence: high.)

### [LOW] MedQuAD license handling is hand-waved ('treat as CC-BY-4.0 per NIH source, or use lavita/MedQuAD if the grader wants a card'). MedQuAD's original NIH sources had per-source usage restrictions (some NIH pages were removed from the original release precisely over rights), so blanket 'treat as CC-BY-4.0' is not safe.
- **Why it matters:** On a project graded partly on GDPR/data governance, an unverified license assertion on a redistributed dataset is a second, smaller version of the MIETIC problem. 'Treat as' is not a license.
- **Fix:** Use a HF copy that ships an explicit license/dataset card and cite that exact card; record the *actual stated* license in your provenance card rather than asserting one. If no clean card exists, downscope EN-from-MedQuAD and lean more on FrenchMedMCQA (Apache-2.0, unambiguous). (Confidence: medium.)

### [LOW] The JAMA framing is miscited and over-read. Verified: it's Goh et al., JAMA Network Open 2024 (not 'JAMA 2023'), and the finding is 'GPT-4 did not significantly improve physician diagnostic accuracy' — NOT a clean 'the bottleneck is human-AI collaboration.' The plan elevates this to a report centerpiece ('assistant value is in guiding, not raw capability').
- **Why it matters:** A graded report that miscites year/journal and over-interprets its flagship citation undercuts credibility on exactly the dimension (rigor) it's projecting. The honest finding actually *strengthens* the POC framing if cited correctly.
- **Fix:** Cite precisely: Goh et al., 'Large Language Model Influence on Diagnostic Reasoning: A Randomized Clinical Trial', JAMA Network Open 2024;7(10). State the actual finding ('no significant improvement in diagnostic accuracy when physicians used GPT-4') and draw the modest, supported inference: 'capability alone is not the bottleneck; integration and over-reliance are real risks — hence our assistant-with-disclaimers framing.' One accurate sentence beats a punchy misquote. (Confidence: high.)

## Reviewer 3 — technical correctness

**Verdict:** The plan is well-scoped, honest, and mostly sound for a 2-week POC — the strategic decisions (Unsloth+TRL with TRL fallback, merge-not-adapter, serverless scale-to-zero, DPO with off-the-shelf preference data, reframing triage as QA-with-guidance) are all defensible and proportionate. But it contains several concrete TECHNICAL errors and version-staleness landmines that will break specific steps as written, and one architectural conflation that makes a whole deliverable (CI/CD) partly fictional. The single most important thing to change: stop treating version pins, eos_token, and thinking-mode as solved from memory — the ecosystem moved to transformers v5 / Qwen3.5 by mid-2026, the plan's pins (transformers==4.56.2, trl==0.22.2) are stale and the official Unsloth installer now pulls transformers from GitHub main (a moving target), and the asserted eos_token=<|im_end|> is wrong for Qwen3-1.7B-Base (it is <|endoftext|>). Fix the install/format facts by copying them live from the official notebook on Day 1, and resolve the GHCR-vs-worker-vllm contradiction so the CI/CD deliverable reflects what is actually deployed.


### [HIGH] WRONG eos_token. The plan states 'set eos_token="<|im_end|>"'. The actual tokenizer_config.json of Qwen/Qwen3-1.7B-Base ships eos_token = "<|endoftext|>" (pad_token also "<|endoftext|>"), tokenizer_class Qwen2Tokenizer. <|im_end|> is the ChatML turn terminator inside the template, not the model's EOS. Hard-coding the wrong eos_token means generation may not stop correctly and loss-masking/labels can be set up around the wrong token.
- **Why it matters:** If you train/serve with the wrong EOS the model can run on past the answer, fail to stop at <|im_end|>, or learn a broken stop signal — directly corrupting the SFT format the whole project depends on. This is a silent, load-bearing bug.
- **Fix:** Do not hard-code either token. After loading the tokenizer, print tokenizer.eos_token and tokenizer.special_tokens_map and use those. For ChatML SFT you typically train so the model emits <|im_end|> at end of the assistant turn and add <|im_end|> to the stop list at inference (vLLM stop=["<|im_end|>"]) while leaving the model's real eos_token alone. Verify with one generation that it stops cleanly. Source: https://huggingface.co/Qwen/Qwen3-1.7B-Base/raw/main/tokenizer_config.json

### [HIGH] Thinking-mode is misunderstood as a train-time-only decision. The plan says: strip <think> blocks during SFT and the model 'won't think', and lists enable_thinking=False as a tokenizer-time flag. Reality: (a) enable_thinking=False must be supplied at REQUEST time via chat_template_kwargs={"enable_thinking": false} (vLLM: --default-chat-template-kwargs '{"enable_thinking": false}' or extra_body per request), and only works cleanly on vLLM >= 0.9.0; (b) when you merge, the base model's chat_template is copied into your model and STILL defaults to thinking-on, so a caller who forgets the flag gets <think> blocks even though you never trained on them. The plan also contradicts itself: it claims starting from Base means 'you own the format with no conflicting alignment', yet relies on the Base model's bundled ChatML+thinking template.
- **Why it matters:** If the FastAPI wrapper or eval harness forgets the flag, the live demo emits <think>...</think> reasoning the grader sees, latency balloons, and the disclaimer/escalation regex checks may miss content buried in reasoning. A 1.7B 'thinking' on triage is exactly the unreliable behavior the plan wants to avoid.
- **Fix:** Make non-thinking the server default: pin vLLM >= 0.9.0 (already in the stack), launch the worker with --default-chat-template-kwargs '{"enable_thinking": false}' (or set it server-side), AND have the FastAPI wrapper always inject chat_template_kwargs={"enable_thinking": false} so callers cannot accidentally re-enable it. Add one eval probe asserting no <think> string appears in output. Sources: https://docs.vllm.ai/en/latest/features/reasoning_outputs/ ; https://github.com/modelscope/ms-swift/issues/5836

### [MEDIUM] DPO-after-merge ordering is implicit and easy to get wrong. The plan's diagram shows SFT LoRA adapter -> DPO on SFT checkpoint -> merge. With ref_model=None, TRL/Unsloth recovers the reference policy by TEMPORARILY DISABLING the adapter — which only works while the model still carries a PEFT adapter. If a learner merges after SFT and then tries DPO with ref_model=None on the merged (adapter-free) model, the implicit-reference trick is undefined and you either silently train against the wrong reference or must load a separate ref model (VRAM blowout on a T4).
- **Why it matters:** Getting the order wrong produces a DPO run that looks like it works (loss moves) but optimizes against a degenerate reference, or OOMs the T4. Both waste a scarce GPU session.
- **Fix:** State the invariant explicitly in the plan: run DPO on the SFT model WITH the LoRA adapter still attached (continue PEFT), ref_model=None, then merge ONCE at the very end (after DPO). Add a one-line assert that the model is a PeftModel before DPOTrainer init.
